## 前言

最近，EJ Holmes 写了一篇颇具争议的文章《MCP is dead. Long live the CLI》，直指 MCP（Model Context Protocol）已经走向消亡。这篇文章在 Hacker News 上引发了 218 条评论的热烈讨论。

作为开发者，我读完全文后发现，作者的观点虽然激进，但切中了一些实际问题。今天这篇文章让我重新思考：**在 AI 辅助开发的时代，我们到底需要什么样的工具接口？**

## MCP 的承诺与现实

当 Anthropic 宣布 MCP 时，行业集体"失去理智"。各家公司争先恐后地部署 MCP 服务器，只为证明自己是"AI 原生"。大量资源涌入新端点、新协议、新授权方案——所有这些都是为了让 LLM 能与它们已经能对话的服务进行对话。

坦白说，我一开始也没完全理解 MCP 的需求。LLM 擅长什么？给它们一个 CLI 和一些文档，它们就能干活了。

MCP 承诺更清洁的接口，但实践中我发现自己还是要写同样的文档：每个工具做什么、接受什么参数，最重要的是什么时候用它们。LLM 并不需要一个新协议。

## CLI 的永恒价值

CLI 是为人类设计的，但对机器同样有效。当 Claude 在 Jira 上做了一些意想不到的操作时，我可以运行同样的 `jira issue view` 命令，看到它确切看到了什么。同样的输入，同样的输出，没有神秘感。

用 MCP，工具只存在于 LLM 会话中。出问题时，我不再是直接运行命令，而是在 JSON 传输日志中摸索。调试不应该需要协议解码器。

## 可组合性：CLI 的绝对优势

这里差异变得巨大。CLI 可以组合。我可以通过管道传递给 jq，用 grep 链接，重定向到文件。这不只是方便，往往是唯一实用的方法。

考虑分析大型 Terraform 计划：

```bash
terraform show -json plan.out | jq '[.resource_changes[] | select(.change.actions[0] == "no-op") | not)] | length'
```

用 MCP，你的选择是将整个计划转储到上下文窗口（昂贵且往往不可能），或者在 MCP 服务器本身构建自定义过滤。无论哪种方式，你都在做更多工作却得到更差的结果。CLI 方案使用已经存在、文档完善、人类和代理都理解得很好的工具。

## 认证已经工作

MCP 在认证上不必要的固执。为什么给 LLM 工具使用的协议需要关心认证？

CLI 工具不在乎。aws 使用配置文件和 SSO。gh 使用 `gh auth login`。kubectl 使用 kubeconfig。这些是经过实战验证的认证流程，无论我在键盘前还是在 Claude 驱动时都同样有效。认证坏了，我修复方式与永远相同：`aws sso login`，`gh auth refresh`。不需要 MCP 专用故障排除。

## 无状态优势：CLI 的根本优势

本地 MCP 服务器是进程。它们需要启动、保持运行、不能静默挂起。在 Claude Code 中，它们作为子进程生成，运行正常，直到不工作为止。

CLI 工具只是磁盘上的二进制文件。没有后台进程，没有状态要管理，没有初始化仪式。需要时它们就在，不需要时它们就消失。

## 实际痛点

除了设计哲学，MCP 还有日常摩擦：

**初始化不稳定：**我已经数不清因为 MCP 服务器没启动而重启 Claude Code 的次数了。有时重试能用，有时我需要清理状态重新开始。

**重复认证无休止：**使用多个 MCP 工具？享受每个的认证乐趣。带 SSO 或长期凭证的 CLI 根本不存在这个问题。认证一次就完成了。

**权限全有或全无：**Claude Code 允许按名称允许列表 MCP 工具，但这仅此而已。你无法限制为只读操作或限制参数。用 CLI，我可以允许列表 `gh pr view` 但要求批准 `gh pr merge`。这种粒度很重要。

## 什么时候 MCP 真的有意义？

我不是说 MCP 完全没用。如果一个工具真正没有 CLI 等价物，MCP 可能是正确的选择。我在日常中仍大量使用，当它是唯一可用选项时。

我甚至可能论证，标准化接口有某些价值，可能存在比 CLI 更合理的用例。

但对于绝大多数工作，CLI 更简单、更易调试、更可靠。

## 真正的教训

最好的工具是同时适用于人类和机器的工具。CLI 有数十年的设计迭代。它们可组合、可调试，并且可以依托已经存在的认证系统。

MCP 试图构建更好的抽象。结果发现我们已经有了一个相当不错的接口了。

## 给构建者的建议

如果你是一家正在投资 MCP 服务器的公司，但你没有官方 CLI，停下来重新思考你在做什么。先发布好的 API，再发布好的 CLI。代理们会自己弄明白的。

## 结语

这篇文章的观点可能过于激进，但它揭示了一个重要趋势：**不是所有新技术都是进步**。

MCP 有它的场景——特别是对于没有 CLI 等价物的新工具。但在绝大多数情况下，成熟的 CLI 仍然是更简单、更可靠的选择。

也许我们需要的不是"AI 原生"协议，而是让已有工具与 LLM 更好协作的改进——无论这是通过 CLI、API 还是其他机制实现。

作为开发者，我应该关注工具的实际价值，而不是被营销术语左右。
