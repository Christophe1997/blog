MCP vs CLI：何时该用什么协议？

前言 最近，EJ Holmes 写了一篇颇具争议的文章《MCP is dead. Long live the CLI》，直指 MCP（Model Context Protocol）已经走向消亡。这篇文章在 Hacker News 上引发了 218 条评论的热烈讨论。 作为开发者，我读完全文后发现，作者的观点虽然激进，但切中了一些实际问题。今天这篇文章让我重新思考：在 AI 辅助开发的时代，我们到底需要什么样的工具接口？ MCP 的承诺与现实 当 Anthropic 宣布 MCP 时，行业集体"失去理智"。各家公司争先恐后地部署 MCP 服务器，只为证明自己是"AI 原生"。大量资源涌入新端点、新协议、新授权方案——所有这些都是为了让 LLM 能与它们已经能对话的服务进行对话。 坦白说，我一开始也没完全理解 MCP 的需求。LLM 擅长什么？给它们一个 CLI 和一些文档，它们就能干活了。 MCP 承诺更清洁的接口，但实践中我发现自己还是要写同样的文档：每个工具做什么、接受什么参数，最重要的是什么时候用它们。LLM 并不需要一个新协议。 CLI 的永恒价值 CLI 是为人类设计的，但对机器同样有效。当 Claude 在 Jira 上做了一些意想不到的操作时，我可以运行同样的 jira issue view 命令，看到它确切看到了什么。同样的输入，同样的输出，没有神秘感。 用 MCP，工具只存在于 LLM 会话中。出问题时，我不再是直接运行命令，而是在 JSON 传输日志中摸索。调试不应该需要协议解码器。 可组合性：CLI 的绝对优势 这里差异变得巨大。CLI 可以组合。我可以通过管道传递给 jq，用 grep 链接，重定向到文件。这不只是方便，往往是唯一实用的方法。 考虑分析大型 Terraform 计划： ...

使用 mcporter 发现和管理 MCP 工具

最近在探索 AI 工具生态时，发现了一个很有用的工具——mcporter。这是一个专门用于 Model Context Protocol (MCP) 的 CLI 工具和生成器，可以帮助我们更方便地发现和使用各种 MCP 服务器提供的工具。 什么是 MCP？ Model Context Protocol (MCP) 是一个开放标准，定义了 AI 助手如何与外部工具和服务进行通信。通过 MCP，我们可以将各种功能集成到 AI 助手中，比如文档查询、数据库访问、API 调用等。 mcporter 的核心功能 mcporter 提供了几个关键功能： 1. 服务器管理 1 2 mcporter list # 列出所有配置的 MCP 服务器 mcporter list <server> --schema # 查看特定服务器的工具定义 这个功能让我能够快速了解有哪些可用的工具，以及每个工具的输入输出格式。 2. 工具调用 1 mcporter call <selector> [key=value ...] 可以直接调用 MCP 工具，支持通过 HTTP URL 或服务器名.工具名的选择器来定位。 3. 配置管理 mcporter 会自动从 config/mcporter.json 加载服务器配置，也支持从编辑器（如 Cursor、Claude）导入配置。 实际应用场景 配置好 mcporter 后，我发现它在以下几个场景特别有用： ...
